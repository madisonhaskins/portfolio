Flex - Case Study

Overview

How can you enter the crowded marketplace of cloud collaboration? Flex was created as a new player in the game that offers what others don’t. The solution was created for a client that knew they wanted to enter this space but didn’t know what direction to go in. After a lot of research and iterations Flex became the final solution.

Problem

The problem the client originally described was that there was room for the cloud collaboration arena to grow and they wanted to be a contributor. They knew the kind of features the solution should have like peer-to-peer collaboration, storage of documents, and the organization of documents and ideas. They didn’t know their target market, they didn’t know the branding or the name, but they knew the end product which is what we strive to give them.

Solution

The final solution is shown below. After conducting user research it was determined that most of the users have used a cloud collaboration platforms before. Knowing this we knew what the user was expecting when approaching a new product in this arena. Using an analysis of the competition it was determined that there were tools that allowed you to collect tags and lists (Evernote), there was a platform that was a good place to store and share files (Google Drive), and there was a tool used to organize things in shared groups but there was not a tool that combined them all into one. This is the solution Flex was designed as, as a one-stop-shop for cloud storage and collaboration.



Research

Because there was already a lot of unknowns communicated in the client brief we needed to get a good feeling of how people already felt about cloud collaboration platforms and storage. After polling around 15 users some conclusions were easy to draw. Our results spoke to the face that people are already using the cloud, and using the cloud to collaborate. They are also already sending files to other people (mostly through email). What this is telling me is that we do not need to reinvent the wheel but design a product with what the users already expect plus some features that are unique to our product. Also using the surface we had the users rank their most desired features that we pulled from the client brief and they were ranked in the order shown above. Using this we can start to formulate user stores and user flows.

  1. 1. Uploading files from computer or phone
  2. 2. Collaboration in real time
  3. 3. Sharing files and groups with others
  4. 4. Creating content like documents and spreadsheet
  5. 5. Saving content found on the web
  6. 6. Organizing/grouping by tags and labels

Competitve Analysis

For my competitive analysis I chose 3 competitors that all used cloud collaboration in different ways. Trello is a small-project management platform that can be used to share files and links and lists with others to collaborate on. Evernote is more of personal organization tool that is available via the cloud on all of your devices. You can keep track of lists, files, recordings, notes and you can also share these with others but it is not widely used for collaboration. Finally, Google Drive is the main player in this arena. It is a document storage and collaboration platform but ictt is limited in that you do not store you ideas, or links in this platform. It became apparent that this is a crowded space and in order to enter it you need a unique niche. I determined that I would combine aspects of all of these competitors and offer something that unifies them into one solution.



User Personas

The client brief did not specify their target audience so some ideal users were identified. WHen i did user surveys I had 2 main types of responders: project managers, and students. This pattern in my user surveys inspired my user personas. The first persona is a project manager who is motivated by organization and collaboration. He is balancing several projects and would like to take more on as long as he can efficiently manage his current projects well. The next user is a nursing student. She is typically working in groups to collaborate on projects and notes. She is motivated by keeping her assignments, links and fellow students organized by class or semester. These our our target users who we are to keep in mind when defining processes and design going forward.



User Stories

Using our research we needed to determine what features are most important. We ranked the highest desirables with insight from the survey and the client brief. Using the feedback from the user survey ranking, the highest ranked were put into the highest priority ranking for the user stories. Those ranked medium and low priority are those supporting actions that users would want to make and that they would expect to make because of their familiarity with the other cloud collaboration platforms.

Design


Wireframes

The first wireframes created were in the form of sketches of the interfaces required to accomplish the high priority user stories. Once the sketches were completed I turned the sketches into low fidelity wireframes that showed how the elements would be positioned on the page but not much more than that. This went through the first iteration of user testing where users were instructed to click through processes like creating an account, creating content, and uploading content.





The feedback from this first user testing session was that:

  1. - "The app looked similar to cloud collaboration platforms they were already familiar with."
  2. - "There were a lot of buttons with ‘+ that all did the same thing’. All 3 users used a different button to add content."
  3. - "Menu placement in the middle of the screen was not the best. The user feedback was I don’t want to drag my mouse to the middle of the screen. Put it closer to where I already clicked."

Taking this feedback I went in to start creating mockups of the application.

Branding

Now that we have feedback about the placement of the content it is time to start thinking visually. The first visual design I did was for the logo. The logo went through several iterations. When I originally did my mind mapping I was coming up with words from nature like ‘lightning’ and ‘power’, and ‘cloud’. My first logo used the nature theme and became a cloud with a lightning bolt. I quickly realized this would be a hard logo to make smaller so I needed to simply. I looked back at my mind map and there were words like ‘speed’ and ‘flexibility’. My initial thought was the shoe with the wings that Hermès the god wore. The shoe icon I created fit really well on the right of the word ‘flex’ which became the name of the tool.

Now that I had a name and a logo the colors and fonts came next. I decided to go with open sans as the body font as it is easy to read. I also used Fjalla as the main font and as the font in the logo because it complemented open sans and looks smart on the page. When I had a more nature theme in mind I used colors like blues and green. When I switched gears I made the blue similar to the DropBox blue and kept the green and added a touch of red.

After many iterations of branding design I was ready to apply it to the wireframes to start created a more high-fidelity mockup.






Prototypes

The high-fidelity mockups went through 3 iterations to finally arrive at the final product. I will highlight the main changes over time alongside the second round of user testing performed.

With the creation of the first iteration of the prototypes I conducted some user testing to determine what kind of design was most favorable. My tests included whether or not I should include a pattern behind the login/onboarding screen, menu placement, and hero image on the landing page. Using the feedback collected from the testing I implemented the most popular choice from all 3.

The first major iteration of design was for the dashboard and the new user dashboard. As you can see below the dashboard layout has stayed relatively the same over time. The main change is from when I did not have content, to where I filled in content. In my second iteration I added a ‘file preview’ but user feedback highlighted that it was too busy and hard to see. In my final design I expanded the file rectangle and added an icon designating what type of file it is to make the page simpler and add more color.



Back when I did my wireframe testing I got some feedback about how when you are on boarded you are brought to a seemingly already full dashboard. When I took this feedback to my mockups I created a blank dashboard that you arrive at after the onboarding process. In my final design you’ll see that I added some words to fill in the empty space with an action button. This was based on user feedback that the screen looked like it was still loading. I also think the addition of some words like ‘Whoops’ to the page gave the project a fun voice.



The last big change that saw some different iterations was the on-boarding and log in pages. The first two iterations never seemed to fit into the overall design. So finally in the final iteration i created a login/onboarding screen that flowed together while also adding some fun verbiage to engage the user and continue to give a voice to the project. The screens works almost as a toggle. If you click the login button it toggles the onboarding button and so on.



Conclusion

After lots of research and iterations of the mockups a final solution emerged. Going back to the original client brief there were a lot of unknowns. After research we determined our approach to the functionality of the product and our target audience. Through iteration of visual design we created a brand and a color scheme and gave the content a voice. Finally after involving really users in the process we arrived at a final design that aligns with the client brief but also what the user wants.
